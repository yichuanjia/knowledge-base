# Memory 对 AI 生成代码的影响对比

| 维度     | 有 Memory                                     | 无 Memory                                     |
|----------|-----------------------------------------------|-----------------------------------------------|
| 命名风格 | 遵循 snake_case，与项目风格一致                | camelCase/snake_case 混用，风格不统一         |
| docstring| 使用 Google 风格，摘要+参数齐全                | 缺失或 reST/Google 混用，格式散乱             |
| 日志方式 | logging 模块输出，禁止裸 print()               | 裸 print() 输出，无日志级别区分               |
| 错误处理 | 采用项目约定模式（自定义异常+重试）            | try-except 随意使用，与项目不一致             |
| 文件位置 | 放入正确子目录（pipeline/、workflows/）        | 堆在根目录或随意命名，破坏结构                |

## 结论

Memory（上下文记忆）是保证 AI 生成代码质量与项目一致性的关键能力。有 Memory 时，AI 能够复现项目的命名规范、docstring 风格、日志方式和目录结构，生成代码可以直接融入项目；无 Memory 时，AI 退化为"通用风格"，每次生成都从零开始猜测，导致风格不统一、规范缺失、代码难以维护。在知识库这类多模块协作项目中，Memory 是保障自动化流水线代码可维护性的基石。
